现在评测有两个主入口：

- [eval_from_jsonl.py](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/eval_from_jsonl.py:1)：读取训练时已经保存的 validation/rollout JSONL。
- [eval_from_model.py](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/eval_from_model.py:1)：从模型 checkpoint/HF 目录重新推理，再评测。

**入口一：eval_from_jsonl.py**
完整流程是：

1. 解析参数  
   输入可以是 JSONL 文件、目录或 glob；`--metrics` 决定指标，`--ks` 决定 Pass@K 的 K 列表。

2. 读取 JSONL  
   [load_jsonl_records](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/io_utils.py:97) 会收集所有 `.jsonl`。如果某行没有 `step`，会从文件名如 `12.jsonl` 推断 step。每行应至少有 `input`、`output`，正确性指标需要 `acc`、`score` 或 `reward`。

3. 统一评测聚合  
   进入 [evaluate_records](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/evaluator.py:83)。它先按 `(step, data_source, 题目标识)` 分组。题目标识优先级是 `unique_id`、`prompt_id`、`id`，都没有才退回到 prompt 文本。这样避免不同题目 prompt 文本碰巧相同而被误合并。

4. per-prompt 指标  
   每个题目组内计算：
   - `pass@k` / `first@1`：用 `acc/score/reward >= threshold` 判断正确。
   - `pass@k` 使用无偏估计；若 `k > 当前题样本数`，该项为空。
   - `first@1` 是该题第一条输出是否正确。
   - `distinct-2` 是该题所有输出的 2-gram 去重率。
   - `self-bleu` 至少需要同题 2 条输出。
   - `semantic-cosine` 至少需要同题 2 条输出，并用 embedding 两两余弦均值。

5. summary 聚合  
   每个 `(step, data_source)` 对 per-prompt 指标取简单平均；如果有多个数据集，会额外生成 `AVG_DATASETS` 行，表示各数据集指标的简单均值。

6. 写输出  
   输出 `validation_summary.csv/json/md` 和 `validation_per_prompt.csv`。

**入口二：eval_from_model.py**
完整流程是：

1. 解析参数并决定每题生成数  
   [infer_samples_per_prompt](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/eval_from_model.py:25) 的规则是：只有 `--metrics` 包含 `pass@k` 时，默认每题生成 `max(--ks)` 条；不包含 `pass@k` 时，每题只生成 1 条。也可以用 `--samples-per-prompt` 手动覆盖。

2. 读取验证 parquet  
   [load_eval_dataset](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/inference.py:90) 支持多个 parquet 或 Hydra 风格列表字符串。默认会去重，优先按 `unique_id`，其次 `extra_info.index`，最后 prompt 文本。当前 `test_repeated.parquet` repeat 次数为 1，因此不会实际放大题目数。

3. 规范化 prompt  
   parquet 中的 `prompt` 如果是 chat messages，会保留为消息列表；推理前通过 tokenizer 的 `apply_chat_template(..., add_generation_prompt=True)` 渲染成模型输入文本。

4. 生成输出  
   [generate_outputs](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/inference.py:258) 根据 backend 选择：
   - `auto`：优先 vLLM，不可用则 HF。
   - `vllm`：一次请求里用 `SamplingParams(n=samples_per_prompt, ...)`。
   - `hf`：把 prompt 展开成多条样本，按 batch 生成。
   采样参数包括 `temperature/top_p/top_k/max_new_tokens/seed`。

5. 数学答案验证  
   每条输出用 [verify_math_response](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/math_verify_utils.py:8) 打分。逻辑和训练 reward 对齐：ground truth 外层补 `\boxed{}`，模型输出只解析最后 512 字符，然后用 `math_verify.verify` 判断等价，写入 `score/acc/pred`。

6. 写 rollout JSONL  
   生成的每条样本会保存到 `model_rollout.jsonl`，并写 `model_rollout.meta.json` 记录模型路径、验证文件、metrics、ks、采样参数、seed 等。

7. 复用 JSONL 评测逻辑  
   推理得到的 records 随后直接进入同一个 [evaluate_records](/data/ruanruihan/adaptive exploration reward/verl/recipe/aer/eval/evaluator.py:83)，所以两个入口最终的指标计算和聚合口径是一致的。

核心区别是：`eval_from_jsonl.py` 只评测已有输出；`eval_from_model.py` 先按指定模型和采样参数重新生成输出，再走同一套评测聚合。
